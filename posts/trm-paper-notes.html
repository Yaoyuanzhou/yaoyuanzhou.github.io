一句话总结

这篇论文的核心观点是：Item ID 本质上是孤立符号，难以共享知识，也不利于大模型 scaling；而 Semantic Tokens 位于更稳定、可共享的语义空间，更适合大规模排序模型持续扩张。

核心贡献：提出 TRM 框架，从 协同感知表示学习、混合 tokenization、联合优化 三个层面系统解决“语义 token 替代 item ID 后老 item 性能下降”的问题，并在线上搜索场景验证效果。

背景与动机

推荐系统中的大型排序模型通常把每个 item 视为一个独立的 ID，再为它分配一个独立 embedding。这种方式简单直接，但一旦 item 集合快速演化，就会暴露出四个根本问题：

  1. 冷启动问题：新 item 没有足够点击和交互，embedding 训练不充分。
  2. 知识遗忘：旧 item 退役后，embedding 中承载的知识一起消失，无法迁移给新 item。
  3. 分布不稳定：item ID 集合不断变化，导致 sparse 表分布波动，dense 参数也更难稳定收敛。
  4. 扩展瓶颈：embedding table 随 item 数量线性膨胀，参数巨大，但知识共享能力很差。
论文的问题意识非常明确：不是简单证明“token 比 ID 好”，而是要回答：为什么 token 更适合 scaling？以及为什么已有 semantic token 方法对老 item 会掉点？

作者给出的理论直觉是：Semantic Token 构成一个更封闭、更稳定、更有结构的集合。根据 scaling law，若损失满足

$$ L \propto N^{-\beta} $$

那么更大的 $\beta$ 就意味着更陡的扩展收益曲线。作者认为 token-based 表示具有更小的有效维度和更好的语义平滑性，因此更容易获得更大的 $\beta$。

相关工作如 TIGER、OneRec、SemID 已尝试用 semantic token 替代 item ID，但它们在新 item 上常有优势，在老 item 上却容易掉点。TRM 的真正创新点，在于把这个现象拆开讲清楚，并给出有针对性的解决方案。

方法详解

TRM（Token-based Recommendation Model）由三个模块组成：先学一个“带用户行为语义”的 item 表示，再把它 token 化，同时保住泛化和记忆，最后用判别式 + 生成式联合训练把 token 结构吃干榨尽。

模块 I：Collaborative-aware Representation

目标是学习一个既懂内容语义、又懂用户行为偏好的 item embedding，作为后续 tokenization 的基础。

第一阶段：In-domain Captioning

作者先用短视频领域数据微调多模态大模型（文中用的是 Qwen2.5-VL），让模型学会理解视频帧、标题、ASR 和 OCR，并生成符合该业务域的 caption。损失函数是标准自回归生成目标：

$$ \mathcal{L}_{cap} = -\mathbb{E}_{(V,T)} \sum_{k=1}^{|T|} \log P\bigl(t_k \mid t_1,\ldots,t_{k-1}, V;\, \theta\bigr) $$

第二阶段：Collaborative Alignment

仅靠内容语义还不够，因为推荐系统真正关心的是“用户视角下什么内容相近”。所以第二阶段再引入协同过滤信号，用对比学习把 embedding 朝用户行为空间拉齐：

$$ \mathcal{L}_{align} = -\mathbb{E}_{(a,b)\in\mathcal{P}} \log \frac{\exp(\mathrm{sim}(h_a,h_b)/\tau)}{\sum_{b'\in\mathcal{B}} \exp(\mathrm{sim}(h_a,h_{b'})/\tau)} $$

最终表示学习总损失为：

$$ \mathcal{L}_{rep} = \mathcal{L}_{cap} + \lambda_{align} \cdot \mathcal{L}_{align} $$

直觉例子

视频 A 是“做蛋糕”，视频 B 是“做茶点”。从画面和文本看，它们可能都属于“甜品制作”，语义很近。

为什么要引入协同信号

如果 A 和 B 的受众完全不同，纯内容表示会把它们放太近；加入协同行为后，两者会被拉开，得到更符合推荐目标的 embedding。

模块 II：Hybrid Tokenization

这是本文最核心的创新。作者发现：直接把 embedding 量化成 semantic tokens，确实利于新 item 泛化，但会损伤老 item 的记忆能力。

问题的根因

RQ-Kmeans 等残差量化方法本质上是粗粒度聚类，只能给出类似 $[s_1, s_2, \ldots, s_n]$ 的 token 序列，但这些 token 的简单拼接或求和，并不能表达 token 组合语义。

例如：如果 token_3 表示“蛋糕”，token_16 表示“蜡烛”，那么 $[3,16]$ 的简单组合并不会天然表达“生日”这个语义概念。

TRM 的解决方案：两类 token 同时保留

  • Gen-tokens：由 RQ-Kmeans 生成，总数约 20,480，负责捕获粗粒度语义，适合新 item 与长尾 item 的泛化。
  • Mem-tokens：基于 BPE 生成，总数最多 20M，负责记住高频 token 组合，对老 item 尤其重要。

最终二者通过 Wide & Deep 结构融合：

$$ h_{hybrid} = h_{deep}(\mathrm{gen\text{-}tokens}) + h_{wide}(\mathrm{mem\text{-}tokens}) $$
可以把它理解为：gen-token 负责“这个 item 大概属于哪类内容”，mem-token 负责“这个 item 作为老 item，到底有哪些高频组合模式需要被记住”。前者偏泛化，后者偏记忆。

模块 III：Joint Optimization

如果只把 token 当普通离散特征喂给排序模型，那么 token 的层级结构没有被真正利用。TRM 因此加入生成式目标，让模型显式学习 token 序列中的依赖关系。

判别式目标：预测用户行为

$$ \mathcal{L}_d = \mathbb{E}_{(X_Q,X_I,X_U,Y)}\; \mathrm{BCE}\!\bigl(Y, P(\hat Y \mid X_Q,X_I,X_U;\,\theta_d)\bigr) $$

生成式目标：对正样本 token 序列做 next-token prediction

$$ \mathcal{L}_g = \mathbb{E}_{(X_Q,X_U,Y)}\; \mathbf{1}[Y=1] \cdot \sum_{j=1}^{L} \mathrm{CE}\!\bigl(s_j, P(\hat s_j \mid X_Q,X_U,s_1,\ldots,s_{j-1};\,\theta_g)\bigr) $$

联合目标

$$ \mathcal{L} = \mathcal{L}_d + \lambda \cdot \mathcal{L}_g \qquad (\lambda = 0.1) $$

这里的关键点不是“多加一个 loss”，而是：token 序列本身有层级结构，前几个 token 决定粗类，后几个 token 决定细粒度语义。NTP loss 会强制模型把这种结构学进去，而不仅仅把 token 当离散 ID 使用。

个人理解

这篇论文最强的地方

  1. 把问题拆对了：不是停留在“token 好像更先进”的表面，而是具体指出已有 semantic token 方法为什么会在老 item 上掉点。
  2. 方法设计有对应关系：Collaborative-aware 解决“内容语义不等于用户语义”，Hybrid Tokenization 解决“泛化与记忆冲突”，Joint Optimization 解决“token 结构没有被学到”。
  3. 理论和工程两头都站住了：既给出 scaling law 的理论解释，也做了线上 A/B，说明它不是只在论文里成立。

我觉得还可以追问的点

  1. 链路较重：MLLM 预训练、RQ-Kmeans、BPE、联合训练，这整套流程对工程团队要求很高。
  2. 极新 item 收益仍有限：mem-token 对老 item 提升明显，但对刚上线的新 item 额外帮助不大。
  3. 生成目标只用正样本：负样本 token 结构没有被系统利用，后续可能还有提升空间。
核心 takeaway：Item ID 的问题在于它把每个 item 当成孤立符号；Semantic Token 的优势在于它把 item 放进了一个可共享、可组合、可扩展的语义空间。对大模型来说，后者显然更适合作为长期演进的基础设施。
