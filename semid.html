💡 一句话总结

SemID 把 TIGER 提出的 Semantic ID 体系从"学术召回实验"搬到了 YouTube 亿级视频的生产排序模型中，核心贡献是解决了"8个整数的 SID 如何适配进排序模型 embedding table"这个工程问题，最终用 SPM 子词分组方案在整体 CTR AUC 和冷启动 CTR/1D AUC 上同时超越了随机哈希 ID 基线，且 serving cost 保持不变。

📄 原文：https://arxiv.org/abs/2306.08121

🔗 前驱工作：TIGER（arXiv 2305.05065，NeurIPS 2023）—— SID 的发明者

三句话核心贡献：
① 提出 N-gram 和 SPM 两种将 SID 序列适配进排序模型 embedding table 的方案；
② 在 YouTube 生产排序模型上验证：SPM-SID 整体 CTR AUC 和冷启动 CTR/1D AUC 均超越随机哈希基线；
③ 用 SID 前缀语义层级分析和时间稳定性实验，证明了 SID 在工业场景下的可行性。
🔍 背景与动机

随机哈希 ID 的三大根本问题

工业推荐系统里，每个 item（视频、商品）都有一个 ID，通常的做法是 随机哈希：把字符串 ID 哈希到一个大 embedding table（几千万行）的某一行， 然后直接拿那行向量作为 item 的表示，让模型在训练过程中更新这个向量。

这套方案快捷实用，但存在三个深层问题：

1
随机碰撞无语义

哈希函数把不同视频随机映射到同一行，"篮球教学"和"美食探店"可能共享同一个 embedding 向量，导致 embedding 空间无法形成有意义的语义聚类，模型泛化受限。

2
长尾视频学不到

YouTube 视频数量符合幂律分布：头部少量视频占据大部分曝光，大量长尾视频曝光极少。长尾视频对应的 embedding 行训练次数严重不足，几乎等于随机初始化。

3
新视频冷启动差

一个刚上线的新视频没有任何历史曝光，它的 embedding 是随机初始化值，排序模型对这个视频几乎没有判断能力，冷启动效果差。

直接用 Dense Content Embedding 行不行？

一个直觉的解决方案是：既然随机哈希没有语义，那就换成内容 embedding（比如 Video-BERT 输出的 2048 维向量），让 item 表示本身携带语义。 直接塞进排序模型试试？

结果：整体 CTR AUC 反而下降了。为什么？

原因在于排序模型对 embedding 有两个互相矛盾的需求：

🧠 记忆能力（Memorization）

记住某个具体视频的质量特征（点击率、完播率等），需要 embedding 是可训练的、可随训练数据更新的。

随机哈希 → 每个 ID 对应独立的一行 → 很强的记忆能力。

🌐 泛化能力（Generalization）

对没见过的新视频和长尾视频也能给出合理排序，需要 embedding 携带语义信息，类似内容能共享表示。

Dense Content Embedding → 语义丰富，但固定不可训练 → 泛化好，记忆差。

加深模型（1.5x/2x 层数）可以弥补 dense embedding 缺乏记忆能力的问题，但 serving cost 成倍增加，生产上不可接受。 VideoRec（端到端训练视频编码器）效果更好，但计算成本是 ID 方案的 10–50 倍，同样无法在生产中部署。

SemID 的核心切入点

核心思想： 把 2048 维的 dense content embedding 压缩成 8 个整数（Semantic ID）， 既保留了语义信息，又让排序模型可以给每个整数学习一个 embedding，恢复记忆能力。 最重要的是，前缀相同的 SID 意味着语义相近，使相似内容产生"有意义的碰撞"而非随机碰撞。
为什么"有意义的碰撞"很关键？
随机哈希的碰撞是完全随机的——"科学纪录片"和"美妆教程"可能碰撞到同一行，互相干扰梯度更新。 而 SID 的前缀共享是语义驱动的——前缀相同说明内容 embedding 在 RQ-VAE codebook 空间中接近， 同前缀的视频共享 embedding 行是 有意义的参数共享，有助于长尾视频从相关头部视频中借到知识。
